一些概念

工作区

在电脑中可以看到的目录，也就是我们编写代码后这些代码所在文件目录下。

版本库

工作区有一个隐藏目录.git，这个不算工作区，是Git的版本库

Git的版本库中有很多东西，最重要的有两个。一个是成为stage（或者叫index）的暂存区，还有Git为我们自动创建的第一个分支master，以及指向master的一个指针叫HEAD

  1. 当我们在编写代码或者修改文件时，是在工作区中进行修改的。

  2. 然后当我们执行git add后，是将工作区的修改文件加入到暂存区中。

  3. 然后当我们执行git commit ，是将暂存区中的所有内容提交到当前分支上。

因为我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，所以，现在，git commit就是往master分支上提交更改。

分支

每次提交，Git都会把它们串成一条时间线，这条时间线就是一个分支。在Git中，又一个主分支，叫master。HEAD严格来说不是指向提交，而是指向master，master才是指向提交的，所以HEAD指向的就是当前分支。

一开始的时候，master分支是一条线，Git用master指向最新的提交，在用HEAD指向master，就能确定当前分支，以及当前分支的提交点。

每次提交，master分支都会向前移动一步，这样，随着你不断提交，master分支的线也越来越长。

当我们创建新的分支，例如dev时，Git新建了一个指针叫dev，指向master相同的提交，再把HEAD`指向dev，就表示当前分支在dev上：

创建一个分支是很快的，因为只有增加一个dev指针，然后修改HEAD的指向，工作区的文件没有任何变化。

不过，之后对工作区的修改和提交都是针对dev分支的了，比如新提交一次后，dev指针往前移了一步，而master指针不变。

这个时候如果切换分支到master（修改HEAD指针），那么所做的修改就会消失，因为修改在dev分支上，master上没有。（所以需要将dev合并到master上）

如果我们在dev的工作完成了，就可以把dev合并到master上。Git合并最简单的方法就是直接把master只想dev的当前提交，就完成了合并。然后可以删除dev分支（即删除dev指针）。

注

分支的切换大多是在本地版本库中的，也就是commit之后，进行分支切换等操作。当然远程也是存在分支切换的，因为本地和远程是对应的，但是本地和远程的联系是通过push来联系的，在本地的切换分支的操作是不影响远程的。

在远程创建一个分支mybranch后，可以在本地checkout 出这个分支，也就是在本地建了一个和远程同名的分支mybranch，然后我们在本地这个分支上进行开发，开发结束后，push到远端的mybranch上。

关于refs/for

  • 这个不是git的规则，是gerrit的规则。

  • Branches, remote-tracking branches, and tags等等都是对commite的引用（reference）,引用都以 “refs/……”表示. 比如

    remote branch: origin/git_int(=refs/remotes/origin/git_int)

    local tag: v2.0(=refs/tags/v2.0)

    local branch: git_int(=refs/heads/git_int)…

  • refs/for/mybranch 的意义在于我们提交代码到服务器之后是需要经过code review 之后才能进行merge的，而refs/heads 不需要code review

如果需要code review，直接push会报错

1
2

$git push origin master
# 那么就会有“! [remote rejected] master -> master (prohibited by Gerrit)”的错误信息

而这样push就没有问题

1

$git push origin HEAD:refs/for/mybranch

这个HEAD应该只当前的版本

reset 的几个参数

–mixed

不删除工作区间改动代码，撤销commit，并撤销git add操作。

这个为默认参数，git reset –mixed HEAD^ 和 git reset HEAD^效果一样

–soft

不删除工作区间改动代码，撤销commit，不撤销git add

–hard

删除工作区间改动代码，撤销commit，撤销git add

注意完成这个操作后，就恢复到上一次commit的状态了

修改commit（revert）

git revert -n <版本号> 进行要修改的commit，然后在commit新的commit

revert、restore、reset的区别

  • restore是关于从索引或另一个提交还原工作树中的文件，此命令不会更新你的分支（和另外两个的主要区别）。该命令还可以从另一个提交中还原索引中的文件。（即这个命令主要目的是将工作区的内容还原为某一版本，对于分支是不会有改变的）
  • revert和reset都是对改变的都是分支。commit后的版本库里的东西。

reset

git reset 的作用是修改HEAD的位置，即 将HEAD指向的位置改变为之前存在的某个版本。

适用场景：如果想要恢复到之前的某个提交，且那个版本之后提交的版本我们都不要了，就可以用这种方法。

使用场景

  1. 当我们开发过程中，开发到一半的时候，这个时候来了一个新的紧急的需求或bug，这个时候我们可以通过这个命令，将现在开发的东西放入git栈中，然后开发新的需求/解决bug，在push之后，我们在把它拿出来，继续开发
  2. 当我们开发时忘记新建分支，直接在master上开发的后，这个时候新建分支切换的时候可能会因为冲突无法切换，此时可以通过stash将冲突的更改放入栈中，然后切换分支，然后再将它取出来，此时也会有冲突的文件，这个时候只需要手动修改就好了。然后将文件add到暂存区中，之后就是后续的操作了。

原理

git stash会把所有未提交的修改（包括暂存的和非暂存的：即add的和没add的更新文件，新建的文件不会存入，commit的也不会存入）都保存起来。

stash是本地的，不会通过push上传到server。

默认情况下，git stash会缓存下列文件：

  • 添加到暂存区的修改（staged changes）
  • Git跟踪的但并未添加到暂存区的修改（unstaged changes）

但不会缓存一下文件：

  • 在工作目录中新的文件（untracked files）
  • 被忽略的文件（ignored files）

git stash命令提供了参数用于缓存上面两种类型的文件。使用-u或者--include-untracked可以stash untracked文件。使用-a或者--all命令可以stash当前目录下的所有修改。
